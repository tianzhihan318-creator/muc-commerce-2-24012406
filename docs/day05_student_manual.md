## 一、实验概览

本实验承接第5天上午的数据分析课程。每名学生需要独立使用第4天清洗后的电商用户数据，提出一个明确的业务问题，完成公共指标、单维专题分析和双维交叉分析，并输出可以复核的统计报表和分析结论。

> **最终目标：** 不是把所有字段统计一遍，而是围绕明确问题形成“代码可运行、指标可复核、结论有边界、文件可复用”的个人项目成果。

### 1. 实验目标

完成本实验后，你应当能够：

- 独立读取并验收清洗后的用户数据；
- 正确定义用户数、流失人数、流失率和行为均值等指标；
- 使用`groupby`和`agg`完成单维分组聚合；
- 使用双字段分组完成多维交叉分析；
- 同时使用样本量和比例解释分组结果；
- 输出规范CSV报表并完成回读验证；
- 使用“数据现象—可能解释—验证建议”的结构撰写结论；
- 将Notebook和输出文件提交到个人GitHub仓库。

### 2. 实验环境与文件

| 项目 | 要求 |
|---|---|
| 运行环境 | Python 3、Jupyter Notebook、Pandas、NumPy |
| 输入数据 | `output/day04_project/ecommerce_customer_cleaned.csv` |
| 实验Notebook | `notebooks/day05_pm_student_project.ipynb` |
| 输出目录 | `output/day05_analysis/` |
| 提交位置 | 个人GitHub项目仓库 |

### 3. 必须遵守的规则

- 必须使用第4天清洗后的数据，不得绕回原始数据重新建立另一套口径；
- 一行数据代表一名用户，不是一笔订单；
- `CustomerID`只用于识别和计数，不适合求平均值；
- 每张流失率或占比表必须同时包含用户数；
- `CashbackAmount`是返现金额，不是消费金额或销售额；
- 当前数据没有订单金额和订单日期，不能计算GMV、客单价或时间趋势；
- 分组差异只能说明当前样本中的关联，不能直接证明因果关系；
- 不得删除Notebook中的任务说明、TODO、检查点、结论区和提交检查。

## 二、项目背景与数据理解

假设你是一名电商运营数据分析人员，需要从用户生命周期、订单行为、优惠行为、服务体验、设备和城市等级等角度识别值得进一步关注的用户群体。

### 1. 数据粒度

> **必须记住：** 当前数据一行代表一名用户。`OrderCount`表示该用户的订单次数，但每行不是一笔订单。

数据共有5630行、22列。完成分析前必须确认：

- `CustomerID`没有重复；
- 核心分析字段没有缺失；
- `Churn`只包含0和1；
- 读取的是第4天生成的清洗后数据。

### 2. 核心字段速查

| 字段 | 含义 | 常用分析方式 |
|---|---|---|
| `CustomerID` | 用户唯一标识 | 使用`nunique`统计独立用户数 |
| `Churn` | 是否流失：1=流失，0=未流失 | 使用`sum`统计流失人数，使用`mean`计算流失率 |
| `Tenure` | 用户使用月数 | 均值、中位数或生命周期划分 |
| `TenureGroup` | 用户生命周期分组 | 生命周期比较 |
| `OrderCount` | 用户订单次数 | 均值、中位数和分组均值 |
| `DaySinceLastOrder` | 距上次下单天数 | 用户近期行为比较 |
| `CouponUsed` | 优惠券使用次数 | 优惠使用行为比较 |
| `CashbackAmount` | 返现金额 | 返现行为比较，不代表消费金额 |
| `Complain` | 是否投诉 | 投诉与流失的关联分析 |
| `SatisfactionScore` | 满意度评分 | 服务体验比较 |
| `PreferedOrderCat` | 用户偏好订单品类 | 品类用户画像 |
| `PreferredPaymentMode` | 用户偏好支付方式 | 支付偏好比较 |
| `PreferredLoginDevice` | 偏好登录设备 | 设备行为比较 |
| `CityTier` | 城市等级 | 城市用户画像 |

### 3. 分析边界自检

| 问题 | 当前数据能否完成 | 原因 |
|---|---|---|
| 总体和分组流失率 | 可以 | 有用户级记录和`Churn`标签 |
| 不同群体的平均订单数 | 可以 | 有`OrderCount`字段 |
| 不同群体的平均返现 | 可以 | 有`CashbackAmount`字段 |
| 销售额和客单价 | 不可以 | 缺少订单金额 |
| 日、周、月订单趋势 | 不可以 | 缺少订单日期 |
| 严格意义上的复购率 | 不可以 | 缺少订单明细和首购、复购事件 |
| 投诉是否导致流失 | 不能直接证明 | 分组差异只能说明关联 |

## 三、个人专题选择

每名学生选择一个专题作为单维分析主线。双维分析可以在此基础上增加另一个业务维度。

| 专题 | 推荐字段 | 参考业务问题 |
|---|---|---|
| A 用户生命周期 | `TenureGroup` | 不同生命周期用户的流失和订单行为有何差异？ |
| B 投诉与服务体验 | `Complain`、`SatisfactionScore` | 投诉、满意度与流失存在怎样的关联？ |
| C 品类与订单行为 | `PreferedOrderCat` | 不同偏好品类用户的规模和订单行为有何差异？ |
| D 支付与优惠行为 | `PreferredPaymentMode` | 支付偏好与优惠行为是否存在分组差异？ |
| E 城市与设备行为 | `CityTier`、`PreferredLoginDevice` | 城市、设备与用户活跃或流失有何关联？ |

### 专题选择要求

- 在Notebook中将`TOPIC`填写为A、B、C、D或E；
- 使用与专题匹配的主分组字段；
- 用一个可以比较和回答的完整问题描述专题；
- 不要只写“分析某字段”；
- 不要为了代码简单而选择与业务问题无关的指标。

**合格问题示例：**

> 不同生命周期用户的流失率、平均订单数和平均返现是否存在差异？

**不合格问题示例：**

> 分析`TenureGroup`。

## 五、结论写作与结果记录

### 1. 结论三段式

#### 数据现象

说明：

- 哪个用户群体；
- 用户数是多少；
- 哪个指标；
- 数值是多少；
- 与哪个群体比较。

#### 可能解释

使用以下措辞：

- 当前样本显示……
- ……与……存在关联；
- 可能与……有关；
- 该群体值得进一步关注。

避免使用：

- 必然；
- 一定；
- 完全因为；
- 直接导致；
- 只要……就会……

#### 验证建议

说明还需要：

- 哪些新增字段；
- 哪个时间信息；
- 哪项进一步分组；
- 哪种模型或实验；
- 哪项业务核查。

### 2. 结果记录模板

| 编号 | 数据现象 | 可能解释 | 验证需求或建议 |
|---:|---|---|---|
| 1 | 填写群体、用户数和指标 | 使用有边界的解释 | 填写新增字段或验证方式 |
| 2 | 填写 | 填写 | 填写 |
| 3 | 填写 | 填写 | 填写 |

### 3. 正误表达对比

| 不规范表达 | 规范表达 |
|---|---|
| 投诉导致用户流失 | 当前样本中投诉用户流失率较高，投诉与流失存在关联，仍需进一步验证 |
| 返现低导致流失 | 流失用户平均返现较低，但不能由均值差异直接推导因果关系 |
| 某组流失率最高，所以最重要 | 该组流失率较高，但还需要结合用户数、业务价值和样本稳定性判断优先级 |
| 长期用户不会流失 | 当前样本中长期用户流失率较低，结果可能受样本和标签定义影响 |

## 八、常见问题排查

| 问题 | 可能原因 | 处理方法 |
|---|---|---|
| `FileNotFoundError` | 工作目录不同或数据文件缺失 | 确认仓库中存在`output/day04_project/ecommerce_customer_cleaned.csv` |
| `KeyError` | 字段名拼写错误 | 使用`df.columns.tolist()`检查准确字段名 |
| 检查点0失败 | 姓名或专题未填写 | 填写`STUDENT_NAME`并将`TOPIC`设为A～E之一 |
| 检查点1失败 | `core_cols`不完整 | 补充Notebook要求的核心字段 |
| 分组字段不在结果中 | 分组字段位于索引 | 使用`reset_index()` |
| 分组用户数之和不等于5630 | 分组字段有缺失或分析前进行了筛选 | 检查缺失值和筛选条件 |
| CSV出现`Unnamed`列 | 导出时写入了索引 | 使用`index=False`重新导出 |
| 中文在Excel中乱码 | CSV编码不匹配 | 使用`encoding="utf-8-sig"` |
| 百分比大于100% | 分子和分母口径不一致 | 在同一分组内使用`Churn.mean()` |
| 样本提示检查失败 | 标记规则与用户数不一致 | 用户数少于30标记“小样本” |
| 结果与课堂不同 | 数据路径、筛选条件或清洗版本不同 | 比较文件路径、`shape`和筛选代码 |
